Testing CBRAIN

the full history

Natacha Beck

January 2017

Why?

  • Confidence of the code, for us but also for external collaborator
  • People are more confident to contibut to an open source project with a testing suite

Original setup

  • Start in 2011, using RSpec a behavior-driven development (BDD) framework for Ruby
  • Tarek's presentation about BDD
  • Natacha's presentation about Continuous integration(CI)
  • Documenation available on GitHub

Original setup

  • Run all the testing suite before commiting something
  • Fix buggy test or code if needed
  • Add new tests if new logic added to CBRAIN
  • Complete the testing suite, RCOV was used to monitor the coverage

Original setup

  • Never really follow the previous rules
  • The testing suite was fix not on a regular basis
  • Fixed buggy tests or code occasionally (public release)

Daily run

  • A daily task was add by Pierre to:
    • Run the testing suite on a VM
    • Log the result on Nagios
  • It increase our "focus" on the tests for a while
  • But.. Systematic failures start to happened and once again we lost the "focus"

Improvements

  • Natacha added test for the API (2014)
  • Remy improved the speed of the testing suite (2015)
  • Tristan (A.A) added test for Boutiques/CBRAIN integration (2016)

Travis

  • A hosted, distributed CI service ti build and test projects hosted on GitHub
  • Open source project may be tested at no charge
  • Already used by Loris and BrainBrowser

Travis

  • Configured by added ".travis.yml" file, to the root of GitHub repository
  • This one specifies the programming language
  • the desired building/the testing environment...

Travis

  • If Travis CI has been activated for a repository, GitHub will notify it for every new commit/pull request
  • Can be configured for specific branches
  • It will check the specific branch and run the command in ".travis.yml"
  • That build the software and run the automated test

Travis, 1st version

  • Based on Tristan and Ludovic work
  • This one allow to deploy a full CBRAIN (Portal/Bourreau/DataProvider)
  • Was able to add the logic to run the testing suite via Travis
  • Too complicate use docker-compose build multiple container
  • Build and rebuild some "fix" part of CBRAIN for example ruby and RoR installation
  • Take ~12min

Travis, 2nd version

  • Based on an existing container that contain the portal and only pull the last version of CBRAIN
  • Still compose with a container that contain the DB
  • Take ~6min

Travis and Slack

  • Create a Slack Channel for the notification
  • Add Slack notification for different repo

Travis, 3rd version

  • Add bourreau side test, a.k.a Boutiques/CBRAIN testing suite
  • Take 5min

Travis, 4th version

  • All in one, version
  • Only one container with everything in it included the DB
  • The DB is now seeded in the main container it is no longer an other container
  • Take ~4min

Conclusion

  • Quick to run
  • Simple setup
  • Quick feedback loop
